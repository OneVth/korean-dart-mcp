## 발견 1 (최상위) — ADR-0030 게이트가 폭주를 못 막음

호출 2: 빈 호출 `{}`로 전체 universe 배치 → `mode:"preview"` **미발동** → DART 16,002콜 소진 → 80% 도달 후 checkpoint 중단. **Stage 31·32에서 막았다고 본 폭주가 그대로 재현.**

### 뿌리 (코드 회수 확정 — 빌드 미갱신 아님)

- 게이트 = `scan-execute.ts:828 if (usagePct > 100)`. `scope_confirmed` 분기 살아있음 → Stage 32 빌드 정상 반영. **버그 아님, 설계대로 동작.**
- `usagePct = calculateDailyLimitUsagePct(estimate.total)` = `(estimate.total / DAILY_LIMIT) × 100` (`_lib/scan-helpers.ts:216`, `DAILY_LIMIT=20000`).
- 호출 2 추정 16,002콜 → usagePct ≈ **80** → `80 > 100` 거짓 → 게이트 미발동 → 완주 진입. **코드가 시킨 대로 정확히 동작.**

### 설계 구멍 둘

1. **분모가 하루 총 한도(20,000)지 잔여 한도가 아님.** 오늘 기 사용분 무시 — "이 스캔이 하루치의 몇 %냐"만 봄.
2. **임계가 100%.** "한 스캔이 하루치를 통째로 넘어야" 멈춤. 16,002콜(80%) 폭주는 100% 미만이라 무사통과. **전체 universe(3,967) 스캔조차 한 번에 못 멈추는 기준값.**

### 정의 오류

ADR-0030이 멈춤 대상을 "한도 *초과*(>100%)"로 정의. 그러나 field-test 폭주의 핵심은 한도 초과가 아니라 **"한 스캔 한 방의 절대 규모가 큼"**(16,002콜, 하루 한도 *내*). 막아야 할 대상을 잘못 정의함 — `>100%`로는 영원히 안 잡힘.

### 실행 중 80% 중단 경로 ≠ 대화 루트

16,002콜 후 80% 도달 → checkpoint는 ADR-0014/0015(실행 *중* 한도 도달 사후 복구)지 ADR-0030 대화 루트(사전 견적·분기)가 아님. 사용자는 견적·options를 *한 번도* 못 받음 → 7부 "사람의 결정 기회" 미복원.

## 폐기된 가설 (실측으로 배제)

- `min_opportunity_score=50` 코드 기본 주입 → **틀림.** 코드 기본값 `.default(0)`, client 무단 주입도 이번 재현 안 됨. 그날 1회 변덕으로 결론. 1번 결함 추적 종료.
- composite/opportunity score 전원 0 → 이번 실측에 안 나타남(후보 자체가 없어 점수 관측 불가). 보류 — candidates>0 나오는 케이스 확보 후 재확인.
- 빌드 미갱신 → **배제.** scope_confirmed 분기 + usagePct 계산식이 동작을 완전 설명.

## 다음 (재설계 방향, ADR 급)

ADR-0030 게이트 임계를 "한도 % 초과"에서 **"이 스캔 한 방의 절대 추정 규모"**로 전환 필요. 예: 추정 콜 N 초과 시(또는 universe M 초과 시) 한도 여유와 무관하게 견적 반환. 잔여 한도(분모) 보정도 별건으로 검토. ADR-0030 재개정 또는 신 ADR — 다음 사이클 0단계 회수 후 확정.
