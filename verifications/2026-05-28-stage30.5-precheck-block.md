# Stage 30.5 사전 검증 — pre-check 차단으로 induty merge 무효 발견

- 일자: 2026-05-28
- baseline: `origin/main` = `ae41c647ba5e6abf44b73d51e2b270824b4806a1` (Stage 30.4 매듭)
- 검증 방식: origin fresh clone (`/tmp/`) raw 회수 (학습 #52). Claude Code 보고 + Claude web 발견 교차 확인.
- 결론: 원 Stage 30.5 핵심("user_preference 통합 → universe 축소 → 한도 초과 직접 결판") **미성립 확정**. scope 길 A로 축소.

## 발견

`resolveInput`에 `loadUserPreference()`를 통합해도, scan_execute 자체 pre-check(ADR-0019)가 그 사전에 name-filter universe로 차단하므로 한도 초과 경로에서 merge 코드는 dead.

### 근거

1. `filterUniverse` (`scan-helpers.ts` line 69) = `excluded_name_patterns` 단독. `included/excluded_industries` 분기 0 (ADR-0010 옵션 D — induty 분기는 11단계 stage1 영역).
2. pre-check `filterConfig`에 `resolved.included/excluded_industries` 미전달 → user_preference merge 후에도 `filtered.length` 불변 → 차단 동일.
3. 실측: `estimateApiCalls(3607)` = `32636` calls = `163.2%` (DAILY_LIMIT 20000). `scan-execute-precheck.test.ts` 18단계 사례 일치.
4. `isIndustryMatch`(line 218~230)는 `stage1StaticFilter` loop 내 `extractCompanyMeta`(company.json) 회수 사후 적용 → pre-check 통과(usagePct ≤ 100) 시에만 도달.
5. cold cache: `getCorpMeta`(corp-meta-cache) miss → `null`. induty_code는 company.json 호출 사후만 회수. 한도 초과 시나리오 = cold cache 첫 실행 → pre-check induty 분기 불가 = ADR-0010 over-estimate 문제 그 자체.

### 부수 발견 — 에러 메시지 오안내

`DailyLimitPreCheckError` 메시지 (line 89): `"Narrow universe via included_industries filter (current universe: N)."` — included_industries는 pre-check `filterUniverse`에 미반영이라 universe 변화 0. 안내대로 해도 차단 동일. Stage 30.4.1(한도 메시지) 정정 영역으로 흡수.

## 결판

- **길 A 채택**: Stage 30.5 = "user_preference 실 실행(`stage1StaticFilter`) 반영"으로 목표 축소. pre-check 통과 universe에서 정상 작동.
- 한도 초과 lever = 별 stage. ADR-0010 개정 + 둘 비교: (1) pre-check 2-phase 분할(stage1 선행 → induty 축소 → stage2+ 재추정, cold cache 실효, 단 `CHECKPOINT_THRESHOLD` 단서) / (2) cache warm-up 명령 신설. 후보 1 우선.
- 학습 #53 신설: #44 "호출부 raw"를 merge 대상엔 적용했으나 그 대상을 게이팅하는 상류 pre-check 호출부 회수 누락 → merge target + 차단 상류 게이트까지 raw 회수 필수. (본 entry prompt 작성 세션 직접 책임.)
